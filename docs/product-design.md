## 1. 设计原则

### 1.1 核心理念

Forge 的初衷：**让不懂代码的人，用自然语言描述需求，就能得到生产级代码并上线。**

- **规范即灵魂** — AI 只是执行者，沉淀多年的工程规范才是质量保障的根本
- **信任靠体系** — 风险分级 + AI Review + 四层测试 + 质量门禁 + 灰度发布 + 紧急停止，组合建立信任
- **按角色服务** — 不同角色看到不同的页面集合，各得其所

### 1.2 用户角色

| 角色 | 核心诉求 | 技术水平 |
|------|---------|---------|
| 产品经理/运营 | 提需求 → 看进度 → 看结果 → 确认上线 | 不懂代码 |
| 技术管理者 | 质量兜底、审批高风险变更、管理规范和分支 | 懂技术 |
| 平台管理员 | 系统运维、平台配置、紧急处置 | 懂技术 |

## 2. 信息架构

### 2.1 全局导航结构

采用**项目优先**的导航模式 — 先选项目，再进入项目子页面。

│   ├── 一键接入（OAuth 授权 GitHub/Codeup，同步全部仓库）

│   └── 项目列表（星标置顶 + 分组归类 + 搜索筛选）
│
├── 进入项目后（按角色展示不同页面）
│   │
│   ├── 产品经理视图

│   │   ├── 任务看板
│   │   ├── 变更结果
│   │   ├── 测试报告
│   │   └── 部署环境
│   │
│   ├── 技术管理者视图（包含产品经理所有页面 +）
│   │   ├── 分支管理
│   │   ├── MR 审批
│   │   ├── 质量大盘
│   │   └── 规范配置

### 2.2 项目列表组织方式

用户通过 OAuth 授权代码托管平台后，**一键同步全部仓库**，用户选择关注的项目。

- **星标置顶** — 关注的项目标星，置顶显示
- **分组归类** — 按代码托管平台的 Organization/项目组自动分组
- **搜索筛选** — 支持按名称、技术栈、状态筛选

## 3. 页面详细设计

### 3.1 登录页

**目的**：用户认证入口。

**页面布局**：
- 全屏深色背景 + Aurora 极光动效
- 居中登录卡片（毛玻璃风格，480px 宽）
- 顶部：Forge Logo + 品牌渐变标题
- 表单字段：用户名、密码
- 登录按钮（Primary 发光按钮）
- 底部：版本号

**交互细节**：
- 输入框聚焦时紫色发光边框
- 密码错误时输入框红色抖动 + 错误提示 "用户名或密码错误"
- 登录成功后淡出跳转到项目大厅
- Session 过期时自动跳回登录页，Toast 提示 "登录已过期，请重新登录"

**Phase 1 仅支持账号密码登录。** Phase 2 增加钉钉扫码/OAuth2。

### 3.2 项目大厅

#### 3.2.1 一键接入

**目的**：快速将用户已有的所有项目纳入 Forge 管理。

**交互流程**：
1. 用户点击"接入代码平台"
2. 选择平台（GitHub / Codeup）
3. OAuth 授权
4. 系统自动同步全部仓库列表
5. 用户选择要关注的项目（可批量勾选）
6. 对选中的项目，AI 自动启动项目画像分析（后台异步）

**项目画像分析**（导入时自动触发）：
- 扫描项目结构、技术栈、依赖
- 识别核心模块和架构模式
- 分析数据库 Schema
- 生成项目画像卡片（技术栈、模块结构、代码规模、活跃度）

**后续提需求时**：按需深入读取相关代码，渐进式积累理解。

#### 3.2.2 创建新项目

**目的**：从零孵化一个新项目。

**交互流程**：
1. **选模板**（快速开始）
   - 标准 Java 微服务
   - Vue 3 前端应用
   - 前后端一体化
   - Java SDK / 类库
   - 空白项目
2. **分步配置**（可跳过用默认值）
   - Step 1: 项目名称 + 描述
   - Step 2: 代码托管平台选择（GitHub / Codeup）
   - Step 3: 部署环境选择（ACK / K8s）
   - Step 4: CI/CD 平台选择（云效 / GitHub Actions）
3. **确认并创建**
   - AI 基于模板 + 配置生成脚手架
   - 推送到代码托管平台
   - 自动配置 CI/CD 流水线

### 3.3 产品经理页面

#### 3.3.1 需求对话

**目的**：产品经理用自然语言提需求，AI 澄清后确认执行。

**交互模式**：混合式（对话 + 确认卡片）

**页面布局**：
- 左侧：历史需求列表（按时间倒序）
- 右侧：对话区域

**交互流程**：
1. 用户输入自然语言需求："给优惠券系统加一个按用户等级发放的功能"
2. AI 流式响应，追问澄清问题："需要支持哪些用户等级？发放规则是固定额度还是按比例？"
3. 多轮对话直到 AI 理解需求
4. AI 生成**需求确认卡片**：
   ```
   ┌─────────────────────────────────────┐
   │  需求确认                           │
   │                                     │
   │  📋 需求摘要                        │
   │  为优惠券系统添加按用户等级发放功能    │
   │                                     │
   │  📦 技术任务拆解                     │
   │  1. 新增 user_level 字段到优惠券表   │
   │  2. 新增发放规则配置接口             │
   │  3. 修改发放逻辑，按等级匹配规则     │
   │  4. 新增单元测试 + 接口测试          │
   │                                     │
   │  ⚡ 预估                            │
   │  影响文件: ~8 个                     │
   │  预估 Token 消耗: ~50K              │
   │  预估时间: ~5 分钟                   │
   │  风险等级: 低                        │
   │                                     │
   │  [确认执行]  [修改需求]  [取消]       │
   └─────────────────────────────────────┘
   ```
5. 用户点击操作按钮：
   - **[确认执行]** → 任务开始执行，自动跳转到任务看板
   - **[修改需求]** → 回到对话模式，卡片收起，AI 回复 "好的，请告诉我需要怎么调整"，用户补充修改意见后 AI 重新生成确认卡片
   - **[取消]** → 弹窗二次确认 "确定取消本次需求？"，确认后清空当前对话

#### 3.3.2 任务看板

**目的**：查看所有需求对应的任务状态。

**三级进度视图（可切换）**：

**概览视图（默认）**：
- Kanban 形式，列为任务大阶段：解析中 → 生成中 → 审查中 → 测试中 → 部署中 → 已完成
- 每张卡片显示：需求摘要、风险等级标签、耗时、负责人

**详情视图**：
- 点击卡片展开，看到每个阶段的关键产出：
  - "需求解析完成，拆解为 3 个子任务"
  - "代码生成完成，新增 5 个文件，修改 3 个文件"
  - "AI Review 评分 92 分，无高危问题"
  - "单测通过 12/12，接口测试通过 8/8"

**实时视图**：
- 流式展示 AI 当前正在做什么
- 实时输出日志/代码生成过程
- 可以随时查看 AI 正在分析哪个文件、生成哪段代码

#### 3.3.3 变更结果

**目的**：查看 AI 生成的代码变更，决定是否满意。

**分层展示**：

**第一层 — AI 总结（默认展示）**：
- 自然语言描述："本次变更新增了优惠券按等级发放功能，涉及 3 个接口变更和 1 张表结构修改"
- 信任指标面板：
  - AI Review 评分：92/100
  - 安全扫描：通过
  - 单测覆盖率：85%
  - 接口测试：全部通过
  - 风险等级：低

**第二层 — 变更摘要（点击展开）**：
- 结构化变更列表（非代码 Diff）：
  - 新增接口：POST /api/coupons/rules、GET /api/coupons/rules/{id}
  - 修改接口：POST /api/coupons/distribute（增加 userLevel 参数）
  - 数据库变更：coupon_rule 表新增 user_level_min 字段
  - 影响模块：coupon-service、coupon-api

**第三层 — 代码详情（继续下钻）**：
- 文件树 + 语法高亮代码查看
- AI Diff 视图（Monaco Editor 内联）：逐行变更 + 每段变更附 AI 解释
- 风险标注：高亮可能有问题的代码行
- **"在 IDE 中打开" 按钮**：跳转到 code-server（VS Code Web），在完整 IDE 环境中浏览代码仓库

**code-server (Web IDE) 交互**：
- 入口：变更结果页 / 项目详情页右上角 "在 IDE 中打开" 按钮
- 行为：新标签页打开 code-server，自动定位到对应项目仓库和分支
- 共享实例：平台维护一个 code-server 实例，按 project_id + branch 隔离工作区
- Phase 1 只读模式：用户可浏览代码但不能通过 IDE 直接提交
- 工作区回收：空闲 30 分钟后自动清理临时目录

#### 3.3.4 测试报告

**目的**：查看四层自动化测试的执行结果。

**四层测试展示**：

| 层级 | 内容 | 展示方式 |
|------|------|---------|
| 单元测试 | AI 生成的对应单测 | 通过/失败数、覆盖率百分比 |
| 接口测试 | AI 根据接口定义自动生成的 API 测试 | 接口列表 + 各场景（正常/异常/边界）结果 |
| 集成测试 | 跨服务的业务流程测试 | 流程图形式展示，每个节点标记通过/失败 |
| 回归测试 | 已有测试用例的全量运行 | 通过率趋势图、失败用例列表 |

每层可展开查看详细用例和失败原因。

AI 根据目标项目技术栈自动选择原生测试工具：
- 单元测试：go test / JUnit 5 / Jest（随项目语言）
- API 测试：httptest / MockMvc / supertest（随项目框架）
- 集成测试：pytest + Playwright（统一 Python）
- 回归测试：CI/CD 流水线中自动触发全量已有测试

#### 3.3.5 部署环境

**目的**：查看各环境的部署状态和发布记录。

**页面内容**：
- 环境卡片：dev / staging / prod，每个显示当前版本、部署时间、健康状态
- 发布记录时间线：每次部署的版本、触发方式（自动/手动）、状态
- 临时环境列表：AI 分支的预览环境，带自动销毁倒计时

### 3.4 技术管理者额外页面

#### 3.4.1 分支管理

**目的**：查看和管理所有 AI 工作分支的状态。

**页面内容**：
- 活跃分支列表：分支名、关联需求、创建时间、最后提交、状态
- 冲突检测面板：哪些分支之间存在冲突、冲突文件列表
- 合并记录：已合并分支的历史，谁审批的、什么时候合并的
- 操作：手动触发合并、解决冲突、关闭废弃分支

**自动化**：
- AI 分支工作完成后自动创建 MR
- 低风险 MR 自动合并（需满足：Review 评分 ≥ 90 + 所有测试通过 + 无冲突）
- 高风险 MR 等待技术管理者审批
- 冲突时 AI 先尝试自动解决（最多 1 次），失败则通知技术管理者

#### 3.4.2 MR 审批

**目的**：审批高风险变更。

**页面内容**：
- 待审批 MR 列表（按风险等级排序）
- MR 详情：
  - AI Review 完整报告（评分 + 问题列表 + 修复建议）
  - 风险评估详情（为什么被判定为高风险）
  - 代码 Diff（带 AI 逐行注释）
  - 四层测试结果
- 操作：批准合并 / 驳回 / 要求 AI 修改 / 添加评论

#### 3.4.3 质量大盘

**目的**：从全局视角监控 AI 生成代码的质量。

**指标面板**：
- AI 生成代码合规率趋势
- AI Review 通过率（首次通过 vs 修复后通过）
- 自动合并代码的线上故障率
- 测试覆盖率趋势
- Prompt 生成质量评分
- 任务成功率（完成 vs 失败 vs 升级人工）

#### 3.4.4 规范配置

**目的**：配置项目级的编码规范和质量规则。

**页面内容**：
- 规范继承链展示：公司级 → 团队级 → 项目级（当前生效的规范）
- 可覆盖项：编码规范选项、Review 规则开关、安全扫描级别
- 风险阈值配置：自动合并的 Review 评分阈值、影响文件数阈值
- Prompt 模板：查看和微调项目使用的 Prompt 模板

#### 3.6.1 代码浏览器（S15）

**目的**：关联 GitHub 后，用户无需离开 Forge 即可浏览仓库完整代码，与任务、变更、PR 深度联动。

**页面布局**：
- 左侧面板：文件树导航（展开/折叠，图标区分文件类型）
- 右侧面板：文件内容展示（Shiki 语法高亮，行号，选中行高亮）
- 顶部工具栏：分支选择下拉框 + 路径面包屑 + 搜索按钮

**文件树导航**：
- 目录结构从 GitHub API 获取（带缓存 + 异步 git pull 保持最新）
- 文件图标按类型区分（.go/.ts/.py/.json 等）
- 点击文件 → 右侧展示内容；点击目录 → 展开/折叠
- 根目录显示 README 摘要（如果存在）

**代码展示**：
- Shiki 语法高亮（支持所有主流语言）
- 行号 + 点击行号可选中行并更新 URL hash
- 大文件截断提示（> 500KB 显示"文件过大，请在 IDE 中查看"）
- 二进制文件显示类型 + 大小信息，不尝试渲染

**分支管理**：
- 下拉切换分支（显示分支名 + 最后提交时间）
- AI 分支高亮标记（`feature/` 前缀，紫色标签）
- 默认展示 `main` 分支

**PR/MR 列表**：
- Tab 切换到 PR 视图
- 显示：PR 标题、状态（Open/Merged/Closed）、作者、创建时间
- 点击 PR → 展示 Diff 视图 + Review 评论
- AI 创建的 PR 标记 "AI Generated" 紫色徽章

**交互细节**：
- 文件树懒加载（只展开当前层级，点击展开子目录）
- 代码内容请求带 loading skeleton
- 404 文件/分支显示空状态 + "分支可能已被删除" 提示

#### 3.6.2 项目画像（S16）

**目的**：展示 AI 自动扫描生成的项目结构化知识，帮助用户和 AI 理解项目全貌。

**页面布局**：
- 顶部：画像概览卡片行（技术栈、API 数量、表数量、模块数、最后扫描时间）
- 主区域：Tab 切换不同画像维度
- 右上角：「重新扫描」按钮

**Tab 页签**：

| Tab | 内容 | 展示方式 |
|-----|------|---------|
| API 清单 | 路由路径、HTTP 方法、请求/响应参数 | 可搜索的表格，点击展开详情 |
| 数据模型 | 表名、字段、类型、索引、关系 | ER 图（简化版）+ 表格列表 |
| 模块结构 | 模块依赖关系、入口文件、职责摘要 | 树形图 / 依赖关系图 |
| 技术栈 | 语言、框架、依赖库、版本 | 卡片分组展示 |
| 业务规则 | AI 提取的隐式业务逻辑 | 列表 + 代码引用链接 |

**扫描状态**：
- 首次扫描：进度条 + 预计时间
- 增量更新：后台自动，右上角显示"画像已是最新"或"正在更新..."
- 扫描失败：错误提示 + 重试按钮

**交互细节**：
- API 清单支持按路径/方法筛选
- 数据模型支持搜索表名/字段名

## 4. AI 工作过程可视化

### 4.1 设计原则

**像看一个员工干活，而不是看一堆日志。**

用户不关心 Prompt、Token、API 调用这些技术细节。他关心的是：
- AI 现在在干嘛？
- AI 做了什么决策？为什么？
- AI 产出了什么？
- 有没有需要我介入的？

### 4.2 页面结构：任务时间线 + 实时工作区

任务详情页采用左右分栏布局：

**左侧 — 任务时间线（步骤流）**

纵向时间线，每个步骤显示：
- 状态图标（✅ 已完成 / 🔄 进行中 / ⬚ 待执行 / ⚠️ 需介入 / ❌ 失败）
- 步骤名称 + 耗时
- 一句话摘要（AI 自动生成的人话总结）
- 关键数据（如"3/5 文件已生成"、"评分 92 分"）

```
✅ 需求理解                         2min
   "理解为：给优惠券系统加用户等级发放功能"

✅ 方案规划                         1min
   "拆解为 3 个子任务，涉及 8 个文件"
   → 点击查看完整方案

⚠️ 方案确认                        等待中
   "AI 决定新建 coupon_rule 表，请确认"
   [同意]  [换个方案]

🔄 代码生成                         3/5 文件
   "正在生成 CouponRuleService.java..."

⬚ AI 审查                          待执行
⬚ 测试生成与执行                    待执行
⬚ 流水线构建                        待执行
⬚ 部署                             待执行
```

**右侧 — 实时工作区**

根据当前活跃步骤动态展示内容：

| 当前步骤 | 实时工作区展示 |
|---------|--------------|
| 需求理解 | AI 的理解摘要 + 任务拆解卡片 |
| 方案规划 | 方案文档（涉及模块、接口变更、数据库变更） |
| 代码生成 | 流式代码输出（像 Claude 一样逐字出现）+ 左侧文件树标记进度 |
| AI 审查 | 逐条展示审查发现 + 实时评分变化 |
| 测试执行 | 四层测试进度条 + 实时结果（通过/失败） |
| 流水线构建 | 构建日志流式输出 |
| 部署 | 部署进度 + 环境状态变化 |

### 4.3 关键交互机制

**决策卡片**：
AI 做重要决策时（如新建表 vs 加字段、选择实现方案、判定风险等级），在时间线上插入决策卡片，展示决策内容和原因。低风险决策自动通过，高风险决策阻塞等待用户确认。

**介入点**：
时间线上出现橙色"需介入"卡片的场景：
- AI 需要用户确认方案（有多个可选方案时）
- 风险评估为高风险，需要确认是否继续
- AI 修复 3 轮仍未通过，升级人工
- 代码冲突 AI 无法自动解决

**回看能力**：
- 历史任务的时间线完整保留
- 点击任意步骤可查看当时的完整产出物
- 可对比不同任务的执行过程

### 4.4 按角色分层展示

**产品经理看到的**：
- 步骤流 + 人话摘要
- 决策卡片 + 介入点
- 每个步骤的产出物（方案文档、变更摘要、测试结果、部署状态）
- 代码生成的流式输出（可选查看）

**技术管理者额外看到的（折叠区域，点击展开）**：
- 每个步骤的 Token 消耗明细
- AI 与模型的原始对话记录（System Prompt + User Input + Response）
- 工具调用详情（读了哪些文件、执行了什么 Git/Shell 命令、调用了哪些外部 API）
- 中间产物的原始数据（JSON 格式的任务拆解、接口契约定义等）

## 6. 自动化测试方案

> **设计决策**: AI 生成测试 + 原生框架运行，不依赖外部测试平台。详见[技术设计文档](technical-design.md) §6 及 §13.6。

### 6.1 设计原则

- **原生框架优先** — 测试代码与业务代码同仓库、同语言、同构建工具，深度绑定
- **AI 生成 + AI 修复** — Claude TestGen Agent 生成测试，失败时自动分析并修复（最多 3 轮）
- **K8s Job 隔离执行** — 每次测试在独立容器中运行，环境一致性有保障
- **无外部测试平台依赖** — 不使用 MeterSphere 等外部平台，减少运维负担

### 6.2 四层测试工具映射

| 测试层 | 工具 | 触发方式 | 结果展示 |
|--------|------|---------|---------|
| 单元测试 | go test / JUnit 5 / Jest（随项目技术栈） | 构建时自动运行 | Forge 测试报告页 |
| API 测试 | httptest / MockMvc / supertest（项目原生框架） | AI 生成后在 K8s Job 中执行 | Forge 测试报告页 |
| 集成测试 | pytest（跨服务链路测试） | AI 编排场景后在 K8s Job 中执行 | Forge 测试报告页 |
| 回归测试 | CI/CD 流水线触发全量原有测试 | 代码合并前自动运行 | Forge 测试报告页 |

### 6.3 测试执行流程

```
AI 代码生成完成
  │
  ├── AI 生成单元/API 测试 → 写入项目代码仓库 → 构建时运行
  │
  ├── AI 生成集成/E2E 测试 → K8s Job 容器中执行（30min 超时）
  │
  ├── 测试结果写入 PostgreSQL → Forge 前端展示四层报告
  │
  └── 失败时 AI 分析日志 → 自动修复 → 重新执行（最多 3 轮）→ 超限升级人工
```

### 6.4 覆盖率与质量门禁

| 指标 | 阈值 | 说明 |
|------|------|------|
| 单元测试覆盖率 | >= 60%（可配置） | 由原生覆盖率工具采集（go tool cover / JaCoCo / c8） |
| API 测试通过率 | 100% | 所有生成的 API 测试必须通过 |
| AI Review 评分 | >= 90 | Claude Reviewer Agent 评分 |
| 安全扫描 | 0 高危 | Semgrep / gosec / eslint-plugin-security |

### 7.1 设计原则

**永不静默失败。** 每个错误状态必须包含：发生了什么、为什么、已完成的工作、下一步建议。

### 7.2 通用空状态

| 页面 | 空状态展示 |

### 7.3 错误状态

| 场景 | 展示 |
|------|------|
| OAuth 授权失败 | 弹窗提示 "授权失败，请重试" + 重试按钮 + 手动配置入口 |

| Token 预算耗尽 | 弹窗警告 "Token 预算已用尽" + 当前用量展示 + 联系管理员入口 |

### 7.4 加载状态

## 8. 设计约束与说明

### 8.1 不做的事情

- 不自建 Git 存储引擎（由代码托管平台提供）
- 不自建代码搜索（由 Claude 语义搜索 + GitHub API 提供）
- 不依赖外部测试平台（AI 生成测试 + 原生框架执行）
- 不在 Phase 1 做灰度部署管理 UI

### 8.2 Phase 1 范围

Phase 1（最小闭环）需要实现的页面：
- 登录页（账号密码）
- 项目大厅（一键接入 + 创建新项目 + 项目列表）

## 9. 视觉设计规范

### 9.1 设计基调

**"深空指挥中心"** — Forge 是一个 AI 工作的指挥台，用户在这里下达指令、观察 AI 执行、审阅成果。视觉上要传达：掌控感、未来感、信任感。

参考产品气质：Linear 的克制优雅 + Devin 的过程透明 + Vercel 的极致黑白对比 + Cursor 的 AI 紫色调性。

### 9.2 色彩体系

### 9.3 Aurora 背景

页面背景不是纯黑，而是带有缓慢流动的 Aurora（极光）效果：
- 2-3 个大尺寸模糊渐变色块，低透明度（5-15%），缓慢漂移（20-30秒循环）
- 颜色组合：深紫 + 青色 + 午夜蓝
- 营造"深空"氛围，同时不干扰内容阅读

```css
/* 示意 */
background:
  radial-gradient(ellipse at 20% 50%, rgba(139, 92, 246, 0.08), transparent 50%),
  radial-gradient(ellipse at 80% 20%, rgba(6, 182, 212, 0.06), transparent 50%),
  radial-gradient(ellipse at 50% 80%, rgba(59, 130, 246, 0.05), transparent 50%),
  #050510;
```

### 9.4 排版

**字体栈**：
- UI 文字：Geist Sans（首选）→ Inter → system-ui
- 代码/数据：Geist Mono → JetBrains Mono → monospace
- 中文回退：-apple-system, "PingFang SC", "Microsoft YaHei"

**字号体系**：
```
text-xs:    12px   标签、元数据、时间戳
text-sm:    14px   次要正文、表格单元格
text-base:  16px   主要正文
text-lg:    18px   小标题
text-xl:    20px   区域标题
text-2xl:   24px   页面标题
text-3xl:   30px   大标题/数字指标
text-4xl:   36px   英雄区数字（如 Token 消耗总量）
```

**排版规则**：
- 字重：400 正文 / 500 强调 / 600 标题，几乎不用 700
- 大字号负字距：标题 `-0.02em` ~ `-0.04em`
- 行高：正文 1.6 ~ 1.7，标题 1.2 ~ 1.3
- 数字/ID/时间戳统一用 Mono 字体

### 9.5 组件风格

**卡片/面板**：
```
背景:      Surface-1 (#0F0F1A)
边框:      1px solid rgba(255, 255, 255, 0.06)
圆角:      12px（大卡片）/ 8px（小卡片）
悬停:      边框变亮至 rgba(255, 255, 255, 0.12) + 微弱上移 translateY(-1px)
```

**按钮**：
```
主按钮:    Primary 背景 + 外发光 box-shadow: 0 0 20px rgba(139, 92, 246, 0.3)
次按钮:    透明背景 + Border 边框
危险按钮:  Error 背景 + 红色外发光
圆角:      8px
按下态:    scale(0.97)
```

**输入框**：
```
背景:      #0A0A15
边框:      Border 色
聚焦态:    边框渐变为 Primary + 外发光 Border-glow
圆角:      8px
```

**标签/Badge**：
```

文字:      语义色本身
如成功态:  bg rgba(16, 185, 129, 0.1) + text #10B981
```

**毛玻璃弹窗（Glassmorphism）**：
```

边框:      1px solid rgba(255, 255, 255, 0.08)
顶部高光:  border-top: 1px solid rgba(255, 255, 255, 0.12)
```

### 9.6 图标体系

- 图标库：Lucide Icons
- 尺寸：20x20
- 线宽：1.5px
- 风格：线性描边，不用填充，与 Geist Sans 字体搭配

### 9.7 动效规范

**原则**：快速、流畅、有目的。每个动效都在传达信息，不是装饰。

| 场景 | 动效 | 时长 | 缓动 |
|------|------|------|------|
| 页面切换 | 内容淡入 + 上移 8px，子元素错开 50ms | 200-300ms | ease-out-quint |
| 卡片悬停 | 边框变亮 + 微弱上移 1px | 150ms | ease-out |
| 按钮按下 | scale(0.97) | 100ms | ease-out |
| 弹窗出现 | scale(0.95→1) + opacity(0→1) + 背景模糊渐入 | 200ms | spring |
| 状态切换 | 颜色平滑过渡（如蓝→绿） | 300ms | ease-in-out |
| AI 流式输出 | 逐字出现 + 闪烁光标 | - | linear |
| 加载态 | Skeleton 骨架屏 + 渐变闪光扫过 | 1.5s 循环 | linear |
| Toast 通知 | 从右下滑入 + 进度条收缩 | 300ms 入 / 5s 停留 | ease-out |
| 进度条 | 渐变填充 + 前端发光点 | 平滑过渡 | ease-out |
| 步骤激活 | 脉冲光环 + 图标缩放 | 2s 循环 | ease-in-out |

**禁止**：旋转加载图标（Spinner）、突然闪现、超过 500ms 的 UI 过渡。

### 9.8 AI 专属视觉语言

AI 在系统中是核心角色，需要专属的视觉标识：

**AI 生成内容标记**：
- 左侧 2px 紫色渐变竖线（#8B5CF6 → #06B6D4）
- 背景微弱紫色底色 rgba(139, 92, 246, 0.03)
- "AI" 小标签（紫色 badge）

**AI 流式输出区域**：
- 深色代码编辑器背景（#0A0A12）
- 代码语法高亮：关键字紫色、字符串青色、注释灰色
- 闪烁光标：紫色竖线，0.8s 闪烁周期
- 已完成文件：文件名后加绿色勾

**AI 决策卡片**：
- 紫色渐变顶部边框（2px）
- 左侧 AI 头像/图标
- 决策内容 + 原因说明
- 底部操作按钮（同意/换方案）

**AI 思考态**：
- 3 个紫色圆点呼吸式闪烁（非转圈）
- "AI 正在分析..." 文字 + 当前操作描述

### 9.9 数据可视化

**实时状态指示器**：
- 运行中：绿色脉冲圆点（2s 呼吸动画）
- 等待中：灰色静态圆点
- 错误：红色静态圆点

**进度条**：
- 分段式进度条，每段代表一个步骤
- 已完成段：实色填充
- 进行中段：渐变填充 + 前端发光点移动
- 待执行段：暗色填充

**Token 消耗**：
- 环形进度图（conic-gradient）
- 中心显示数字（Mono 字体）
- 颜色随用量变化：绿→黄→红

**质量评分**：
- 大数字展示（text-4xl Mono）
- 颜色语义：≥90 绿色 / 70-89 黄色 / <70 红色
- 下方小字显示趋势箭头

**时间线**：
- 左侧纵向连接线（Border 色，1px）
- 步骤节点：圆形图标（状态色填充 + 白色图标）
- 活跃步骤：脉冲光环效果
- 内容区在节点右侧展开

### 9.10 响应式

- 桌面优先（1440px 设计基准）
- 最小支持宽度：1280px
- 不需要移动端适配（后续 IM 机器人覆盖移动场景）
