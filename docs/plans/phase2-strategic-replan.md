## 一、战略判断：Forge 在 2026 年 AI 开发工具市场的位置

### 1.1 市场格局速写

2026 年 4 月的 AI 开发工具市场已经分化为四个赛道：

| 赛道 | 代表 | 估值/ARR | 核心能力 |
|------|------|---------|---------|
| **开发者 AI IDE** | Cursor, Copilot, Claude Code, Codex | 数十亿美金 | 开发者在 IDE 里 + AI 辅助 |
| **自主 Agent** | Devin, Factory | 数亿美金 | 从 Ticket 到 PR 全自动 |
| **Vibe Coding** | Lovable, Bolt.new, Replit | Lovable $20M ARR/2个月 | 非技术人员快速出原型 |
| **企业代码治理** | Augment, Tabnine, Sourcegraph | 企业级定价 | 合规、安全、大代码库 |

### 1.2 Forge 不应该打的仗

- **不和 Cursor/Copilot 比代码生成质量** — 他们有数百人团队和数十亿资金，代码生成正在快速商品化
- **不和 Lovable/Bolt 比出原型速度** — 他们的优势是 "5 分钟上线一个 MVP"，Forge 的价值不在速度
- **不和 Devin 比全自主能力** — Devin 从 $500 降到 $20，说明纯自主 Agent 的商业模式遇到了挑战

### 1.3 Forge 应该占领的位置

**"受控的 AI 工程平台"** — 处于 Vibe Coding（快但不安全）和企业治理（安全但需要开发者）之间的空白地带。

核心叙事：**反 Vibe Coding。**

2026 年的行业数据已经证明了 Vibe Coding 的代价：
- AI 生成代码的安全漏洞率是人工代码的 **2.74 倍**
- AI 代码的技术债累积速度是人工代码的 **3 倍**
- Amazon 2026 年 3 月因 AI 辅助代码导致 6 小时宕机，损失约 630 万订单
- CVE 中 AI 相关条目从 1 月的 6 个增长到 3 月的 35+ 个

Forge 的定位：**"让非技术人员也能安全地通过 AI 生成生产级代码"** — 不是 "快速出原型" 而是 "安全上生产"。

### 1.4 三个核心竞争壁垒

| 壁垒 | 说明 | 竞品对标 |
|------|------|---------|
| **规范中心** | 结构化、版本化的工程规范，机械化注入每次生成 | 无直接竞品。Tabnine 有 org policy 但不做 prompt 注入 |
| **约束化流水线** | 4 层测试 + 质量门禁 + AI Review，约束在生成循环内 | Codacy/CodeScene 做门禁但不在生成循环内 |
| **非技术入口 + 企业级输出** | 自然语言输入 → 规范约束 → 生产级代码 | Lovable 做非技术入口但无企业级约束 |

## 二、现状审计：Phase 1 建了什么，实际能力边界在哪

### 2.1 已建成的能力（可演示的完整流程）

```
用户创建任务 → 输入自然语言需求 → AI 对话澄清 → 确认执行
→ AI 规划 DAG 任务分解 → AI 生成测试 → AI 生成代码 → AI Review（3轮修复）
→ 测试执行 → Git 推送 + 创建 PR → SSE 全程实时更新
```

这个 **端到端链路已经跑通**，包括：
- 10 个后端 API 模块全部工作
- 50+ 前端组件，23 个页面
- 6 个 AI Agent（5 个实际运行）
- Temporal 工作流完整编排
- 多模型路由 + 断路器
- K8s Job 执行 + Mock 降级

### 2.2 已建但有关键缺陷的能力

| 能力 | 现状 | 缺陷 |
|------|------|------|
| **需求分析** | 对话 UI 存在，AnalystAgent 代码存在 | 后端返回占位响应，未调用 AI |
| **项目画像** | 画像页面 + 6 维存储 + ProfilerAgent | 触发扫描未接 Temporal 工作流 |
| **规范注入** | 规范 CRUD + 继承链 + effective specs API | 未验证规范是否真正注入到 AI prompt |
| **代码浏览** | 文件树 + Shiki + 分支 + PR + 异步 git | 已基本完成（S15 大部分已实现） |

### 2.3 未建但存在详细计划的能力

- S9 (任务拆分增强) — 已有详细 6-task 计划
- S10 (测试先行) — 已有详细 5-task 计划
- S11-S17 — 有高层描述但无详细执行计划

### 2.4 关键洞察

**Phase 1 建了 "能跑的全链路"，但链路上有几个关键环节是假的：**
1. 需求分析是假的（返回占位文本）
2. 规范注入没有被验证（可能 AI 生成代码时根本没看规范）
3. 项目画像没有被触发（AI 不了解目标项目的真实结构）

**这意味着：当前的代码生成质量可能并不比 Cursor/Copilot 好。** 规范没有注入，画像没有加载，分析没有运行 — 那 Forge 的核心差异化能力（Harness Engineering）实际上还没有生效。

## 三、重新排序：什么才是真正重要的

### 3.1 产品经理视角 — 用户价值分析

假设第一批用户是一个 3 人小团队（1 PM + 1 技术负责人 + 1 运营），他们想用 Forge 给自己的 SaaS 产品加新功能。

**用户旅程中的关键卡点：**

| 步骤 | 期望 | 当前实际 | 影响 |
|------|------|---------|------|
| 1. 描述需求 | AI 聪明地追问和澄清 | AI 返回占位文本 | **致命** — 第一步就崩了 |
| 2. 看到 AI 理解了需求 | 结构化确认卡片 | 跳过分析直接规划 | **严重** — 用户没有信任感 |
| 3. AI 了解我的项目 | 知道项目用什么技术、有哪些 API | 画像未被触发 | **严重** — AI 盲目生成 |
| 4. 看到代码质量 | 遵循我的编码规范 | 规范注入未验证 | **严重** — 核心卖点失效 |
| 5. 看到测试结果 | 4 层测试全部通过 | 测试在 mock 模式 | **中等** — 演示可以 |
| 6. 代码推到我的仓库 | 分支 + PR + Review | 已工作 | ✅ |
| 7. 浏览代码确认 | 文件树 + 高亮 + Diff | 基本完成 | ✅ |
| 8. 部署到我的环境 | K8s 部署 + 预览 | 需要 K8s 集群 | 后置 |

**结论：当前系统的前 4 步（用户建立信任的关键窗口）全部有问题。**

### 3.2 技术大牛视角 — 架构债务分析

| 债务 | 严重性 | 说明 |
|------|--------|------|
| AnalystAgent 未接线 | 🔴 高 | 1 个函数调用就能解决，但被搁置了 |
| 规范注入未验证 | 🔴 高 | 可能 context builder 已经做了，但没有端到端测试证明 |
| 画像扫描未接线 | 🟡 中 | ProfilerAgent 存在，需要接 Temporal |
| JWT 长期 token 问题 | 🟡 中 | ai-worker 的 token 可能过期 |
| qwen-max JSON 不稳定 | 🟢 低 | 降级链会切到其他模型 |

### 3.3 重新排序的核心逻辑

**老计划的问题**：按照流水线阶段顺序 (S9 → S10 → S11 → ... → S17) 线性推进，但忽略了一个事实 — **链路上的关键环节是空的**。

**新策略**：先把已建成的链路做实（填补空洞），再增强流水线各阶段。

```
老策略：S15 → S16 → S9 → S10 → S11 → S12 → S13 → S14 → S17
                                        ↑ 需要 K8s 集群

新策略：
  Phase 2a（补洞 — 让核心链路真正生效）
  ├── SX-1 接线需求分析（AnalystAgent → Temporal → 前端）
  ├── SX-2 验证+修复规范注入（确认 effective specs 真正进入 AI prompt）
  ├── SX-3 接线项目画像扫描（ProfilerAgent → Temporal → 自动触发）
  └── SX-4 端到端质量验证（用真实项目跑一遍完整流程，修复所有断点）

  Phase 2b（增强 — 提升流水线深度）
  ├── S9  任务拆分增强（DAG 可视化）
  ├── S10 测试先行系统
  └── S11 代码生成增强 + Lint

  Phase 2c（基础设施 — 解锁自动化执行）
  ├── Infra-1 最小可用 K8s 环境（本地 k3s 或 ACK 最小集群）
  ├── S12 自动化测试执行（K8s Job）
  └── S13 制品管理（Docker + ACR）

  Phase 2d（部署闭环 — 完成端到端）
  ├── S14 K8s 部署
  ├── S16 项目画像 RAG 增强
  └── S17 云端预览
```

## 四、Phase 2a — 补洞：让核心链路真正生效

> **目标**: 不加新功能，只修复已建成链路中的空洞。完成后，Forge 的核心差异化能力（规范约束 + AI 分析 + 项目理解）首次真正生效。
>
> **预估工期**: 3-4 天
> **交付标准**: 用一个真实 GitHub 项目走完全流程，AI 输出的代码明显体现规范约束和项目理解。

### SX-1: 接线需求分析（1 天）

**问题**: 用户发消息给 AI，后端返回 "AI analysis coming soon" 占位文本。AnalystAgent 代码已存在但未被调用。

**修复方案**:
1. `conversation/service.go` 的 `SendMessage` 方法中，将占位响应替换为 Temporal Activity 调用
2. 调用 `analyze_requirement_activity`，传入用户消息 + 项目上下文
3. AI 返回的分析结果（澄清问题、风险、技术栈检测）写入 conversation 表
4. SSE 推送分析结果到前端
5. 支持多轮：用户回答追问后，再次触发分析，直到 AI 认为需求清晰

**验收**:
- 用户输入 "给用户系统加一个积分功能"
- AI 应该追问（"积分是消费获得还是活动获得？有哪些兑换规则？"）
- 多轮对话后给出结构化需求确认

### SX-2: 验证并修复规范注入（1 天）

**问题**: 规范存储和继承链已实现，effective specs API 已工作。但不确定 AI 生成代码时是否真正使用了这些规范。

**修复方案**:
1. 读 `ai-worker/src/context/builder.py`，验证 `build_context()` 是否调用 effective specs API
2. 如果没有 → 添加 HTTP 调用获取项目 effective specs，注入到 system prompt
3. 添加端到端测试：创建一条 "禁止使用 var 声明变量" 的编码规范 → 触发代码生成 → 验证生成的代码中没有 var
4. 在 AI Review 环节也注入规范作为 Review 标准

**验收**:
- 在规范中心配置 "所有函数必须有错误处理" → AI 生成的代码每个函数都有 error handling
- 修改规范为 "响应必须用 Result 泛型包装" → AI 生成的代码使用 Result 包装

### SX-3: 接线项目画像扫描（1 天）

**问题**: ProfilerAgent 已实现 API 目录扫描、DB Schema 解析、模块依赖图生成。但 `/api/projects/:id/profiles/scan` 端点返回固定响应，未触发 Temporal 工作流。

**修复方案**:
1. 创建 `ProfileScanWorkflow`（参考 TaskWorkflow 模式）
2. 工作流调用 `scan_project_profile_activity`（已在 ai-worker 注册）
3. 扫描结果写入 `project_profiles` 表
4. 前端 "重新扫描" 按钮对接新工作流
5. 项目导入时自动触发首次扫描
6. Context Builder 在构建 AI 上下文时加载项目画像

**验收**:
- 导入一个 Go 项目 → 自动触发画像扫描 → 画像页面显示 API 清单和 DB Schema
- AI 生成代码时的 system prompt 中包含项目画像摘要

### SX-4: 端到端质量验证（1 天）

**问题**: 各模块独立工作，但全流程从未用真实项目端到端验证过。

**修复方案**:
1. 准备一个示范 GitHub 项目（简单 Go API + PostgreSQL）
2. 在 Forge 中导入该项目
3. 执行完整流程：描述需求 → AI 分析 → 确认 → 规划 → 测试生成 → 代码生成 → Review → 推送
4. 记录每个环节的输入输出，修复发现的所有断点
5. 验证：
   - AI 分析是否真正理解了需求？
   - 规划的 DAG 是否合理？
   - 测试代码是否可运行？
   - 生成的代码是否遵守规范？
   - Review 是否能发现真实问题？
   - 推送的分支和 PR 是否完整？

**验收**: 一个完整的从需求到 PR 的录屏/日志，证明全链路真正工作。

## 五、Phase 2b — 增强：提升流水线深度

> **目标**: 在核心链路生效的基础上，增强流水线关键环节的能力。
>
> **预估工期**: 5-7 天
> **交付标准**: DAG 可视化、测试先行、Lint 内联检查全部上线。

### S9: 任务拆分增强（2 天）

*基于已有计划 `docs/plans/S9-task-decomposition.md`*

核心交付：
1. PlannerAgent 输出标准 DAG 结构（节点 + 边 + 并行轨道）
2. 前端 React Flow 可视化 DAG（节点可点击查看详情）
3. 需求 ↔ 任务双向追溯矩阵
4. 工时估算显示在节点上
5. 过大任务自动拆分提示

### S10: 测试先行系统（2-3 天）

*基于已有计划 `docs/plans/S10-test-first.md`*

核心交付：
1. TestWriterAgent 在代码生成前运行
2. 测试用例注入到 CoderAgent 的上下文（"你的代码必须通过这些测试"）
3. 前端显示测试用例预览（用户可在确认前审查）
4. 支持原生框架：go test / Jest / pytest
5. 覆盖率目标可在项目设置中配置

### S11: 代码生成增强 + Lint（2 天）

核心交付：
1. 语言约束强制执行（已部分实现，需验证）
2. 生成后立即运行 Lint（eslint / golangci-lint / ruff），结果注入 Review
3. 自动生成 Dockerfile（已部分实现，需完善多阶段构建）
4. 生成代码的 Diff 预览增强（Monaco Editor）
5. 规范违规自动修复（Review 发现 → 自动重生成 → 最多 3 轮）

## 六、Phase 2c — 基础设施：解锁自动化执行

> **目标**: 搭建最小可用的 K8s 执行环境，让测试和构建在真实容器中运行。
>
> **预估工期**: 5-7 天
> **交付标准**: AI 生成的测试可以在 K8s Job 中真实运行并返回结果。

### 关键决策：K8s 环境选择

| 选项 | 优势 | 劣势 | 成本 |
|------|------|------|------|
| **本地 k3s/minikube** | 零成本，即时可用 | 仅开发用，性能有限 | 免费 |
| **ACK 最小集群（3节点）** | 真实生产环境 | 需要购买，需要时间搭建 | ~¥2000/月 |
| **GitHub Actions + Docker** | 无需自建集群 | 执行受限于 Actions 配额 | 按量 |

**推荐**: 先用 **本地 Docker + k3s** 开发和验证 S12/S13 的逻辑，代码设计支持 K8s 切换。ACK 集群在需要真实多租户部署时再搭建。这样不会被基础设施阻塞。

### Infra-1: 最小 K8s 执行环境（2 天）

1. 本地 k3s 或 Docker-in-Docker 环境搭建
2. forge-task-runner 基础镜像构建（包含 Go/Node/Python 运行时）
3. K8s Job 模板标准化（超时、资源限制、日志采集）
4. 测试连通性：从 forge-core 创建 K8s Job → 执行 → 获取结果

### S12: 自动化测试执行（3 天）

核心交付：
1. AI 生成的测试在 K8s Job 容器中运行
2. 四层顺序执行：单元 → API → 集成 → 回归
3. 上一层失败则阻断（不进入下一层）
4. 测试日志实时推送到前端（SSE）
5. 覆盖率门禁（>= 配置阈值才通过）
6. 失败时 AI 分析日志 → 自动修复 → 重新执行（最多 3 轮）

### S13: 制品管理（2 天）

核心交付：
1. Docker 多阶段构建（AI 生成 Dockerfile）
2. 镜像推送到 ACR / Docker Hub
3. 语义化版本号自动生成
4. 镜像漏洞扫描（Trivy）
5. 制品元数据记录（版本、大小、层数、基础镜像）
6. 前端制品列表 + 详情页增强

## 七、Phase 2d — 部署闭环：完成端到端

> **目标**: 用户在 Forge 中描述需求，代码最终部署到 K8s 环境并可预览。
>
> **预估工期**: 5-7 天（需要 ACK 集群）
> **交付标准**: 低风险任务全自动从需求到部署，高风险任务卡在审批环节。

### S14: K8s 部署（3 天）

核心交付：
1. AI 生成 K8s 资源清单（Deployment / Service / Ingress / ConfigMap）
2. 低风险 → 自动部署到 dev/staging
3. 高风险 → 创建 MR → 等人工审批
4. 部署状态实时推送
5. 回滚能力（一键回到上一版本）
6. 环境管理页面增强（健康状态、资源使用率）

### S16: 项目画像 RAG 增强（2 天）

*在 SX-3 基础画像扫描之上*

核心交付：
1. pgvector 存储画像向量
2. 语义搜索：AI 生成代码时自动检索相关画像片段
3. 增量更新：代码推送后自动触发画像更新
4. 前端画像页面增强（ER 图、模块依赖图可视化）

### S17: 云端预览环境（2 天）

核心交付：
1. AI 检测项目类型 → 自动生成 Dockerfile + K8s 资源
2. PR 创建时自动创建预览环境
3. 预览 URL：`{taskId}.preview.forge.example.com`
4. 空闲 30 分钟自动销毁
5. PR 合并/关闭时自动清理
6. 前端预览环境卡片（状态、URL、日志）

## 十一、一句话总结

**先把已建好的 80% 做到真正能用的 100%（Phase 2a），再增强流水线深度（Phase 2b），最后解锁基础设施和部署闭环（Phase 2c/2d）。**

不要在一个地基有裂缝的楼上继续加盖新层。先补裂缝，再盖楼。

## 附录 A: Phase 2a 后的产品差异化验证

Phase 2a 完成后，Forge 应该能展示这个场景：

```
1. 用户导入一个 Go API 项目（已有数据库和 10 个 API）
2. Forge 自动扫描项目画像（检测到 Go + Gin + PostgreSQL + 15 张表 + 10 个 API）
3. 用户输入："给用户系统加一个积分功能，用户消费获得积分，积分可以兑换优惠券"
4. AI 追问："积分比例是什么？1元=多少积分？积分有有效期吗？兑换优惠券有最低积分要求吗？"
5. 用户回答后，AI 给出结构化需求确认卡片
6. 用户确认 → AI 生成 DAG 任务分解
7. AI 先生成测试用例（go test），再生成代码
8. 生成的代码遵循项目已有的编码规范（用 Result 包装、用构造器注入、错误处理风格一致）
9. AI Review 通过 → 测试运行 → 代码推送到 GitHub → 创建 PR
```

**这个场景是 Lovable/Bolt/Cursor/Copilot 都做不到的。** 它们要么不理解项目现有结构，要么不强制执行编码规范，要么不走测试先行流程。这就是 Forge 的护城河。
