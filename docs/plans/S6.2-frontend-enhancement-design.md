# S6.2 — 前端体验增强设计规格

> **版本**: 1.0
> **日期**: 2026-04-02
> **前置**: S6 (AI Worker), S6.1 (Remediation), product-design.md §4

---

## 1. 目标

将 S6 已有的前端从"能跑通"提升到"能用"：

1. **任务详情页重做** — 按产品设计做左右分栏（时间线 + 实时工作区）
2. **代码预览增强** — Shiki 语法高亮 + 流式代码输出
3. **变更结果页** — 三层展示（AI 总结 → 变更摘要 → Monaco Diff）
4. **测试报告页** — 四层测试结构骨架 + 空状态

**不做**: 流式 AI 文字输出（当前架构不支持）、code-server 集成（S7 范围）、实际 CI/CD 数据、实际 GitHub Diff 数据。

---

## 2. 新增依赖

| 包 | 用途 | 安装方式 |
|---|------|---------|
| `shiki` | VS Code 级语法高亮 | `npm i shiki` |
| `@monaco-editor/react` | Monaco Diff 视图 | `npm i @monaco-editor/react` |

Shiki 按语言懒加载（仅 java, go, python, typescript, javascript, sql, yaml, json, xml, bash）。Monaco 用 `dynamic import` 避免阻塞首屏。

### 3.2 左侧：步骤时间线（增强版）

基于现有 `step-timeline.tsx` 重构为 `StepTimeline`：

- 每步显示：状态图标 + 名称 + 耗时 + **一句话摘要**
- 摘要来源：step.output 解析后提取 summary 字段
  - PLAN: `"{title} — 拆解为 {N} 个任务"`
  - GENERATE: `"生成 {files_changed} 个文件, +{lines_added} -{lines_deleted}"`
  - REVIEW: `"评分 {score}/100, {findings.length} 个问题"`
  - 其他: 固定摘要文字
- 点击步骤 → 右侧工作区切换到该步骤的产出物
- 当前激活步骤（RUNNING）有紫色脉冲动画
- 已完成步骤可点击回看

### 3.3 右侧：实时工作区

新组件 `TaskWorkspace`，根据 `activeStep` 渲染不同内容：

| activeStep | 组件 | 内容 |
|------------|------|------|
| 无/PENDING | 空状态 | "等待任务开始..." 或需求描述 |
| PLAN (completed) | `PlanOutputCard` | 任务拆解列表 + 风险评估 |
| GENERATE (running) | `StreamingCodeView` | 流式代码输出（SSE → Redis → 前端） |
| GENERATE (completed) | `CodePreviewPanel` (增强版) | 文件树 + Shiki 高亮代码 |
| REVIEW (completed) | `ReviewReportCard` | 评分 + findings 列表 + 修复建议 |
| COMPLETE | 完成总结 | 最终统计（文件数、通过率、耗时） |

### 3.4 新组件清单

#### `PlanOutputCard`
- 从 PLAN step output 解析
- 显示：任务标题、任务拆解列表（序号 + 名称 + 涉及文件）、风险等级 badge、风险因素列表

#### `StreamingCodeView`
- 订阅 SSE `code_token` 事件
- 逐 token 追加到 `<pre>` 区域（使用 requestAnimationFrame 合批渲染）
- 底部显示当前模型 + token 计数
- 完成后自动切换到 CodePreviewPanel

#### `ReviewReportCard`
- 从 REVIEW step output 解析
- 顶部：评分环形图 + passed/failed badge
- 中部：findings 表格（severity badge + file + line + message + suggestion）
- 底部：修复建议摘要

## 4. 代码预览增强

### 4.1 Shiki 语法高亮

新组件 `ShikiCodeViewer` 替代现有 `CodeViewer`：

- 使用 `shiki` 的 `codeToHtml()` 渲染带高亮的 HTML
- 主题：`vitesse-dark`（与深空主题协调）
- 语言检测：从文件扩展名自动映射
- 行号显示
- 懒加载：语言 grammar 按需加载

### 4.2 文件树增强

现有 `FileTree` 已可用，小改：
- 文件图标根据语言类型显示不同颜色
- 显示每个文件的行数变化（+N / -N）

### 4.3 流式代码输出

`StreamingCodeView` 组件：
- 数据源：SSE `code_token` 事件（Python AI Worker 通过 Redis publish，Go SSE handler 订阅转发）
- 渲染：`<pre>` 容器 + requestAnimationFrame 合批追加
- 自动滚动到底部
- 代码完成后显示 "生成完毕" + 文件统计

### 5.2 数据来源

Phase 1 的变更数据来自任务的 GENERATE step output（不是真实 GitHub Diff）：
- `files[]` — 文件路径、内容、操作类型（create/modify）
- `commit_message` — 提交信息
- REVIEW step output — 评分、findings

Phase 2/S7 会从 GitHub API 获取真实 PR Diff。

### 5.3 Monaco Diff 视图

- 使用 `@monaco-editor/react` 的 `DiffEditor` 组件
- `original`: 空字符串（create）或 placeholder（modify，Phase 1 无原始代码）
- `modified`: 生成的代码
- 主题：`vs-dark`
- 只读模式
- 按需加载（`dynamic import`）

### 5.4 信任指标面板

4 个指标卡：
1. **AI Review 评分** — 从 REVIEW output.score 获取, 颜色：≥90 绿, ≥70 黄, <70 红
2. **安全扫描** — Phase 1 固定显示 "待接入"（Phase 2 接 Semgrep）
3. **测试覆盖率** — Phase 1 固定显示 "待接入"
4. **风险等级** — 从 PLAN output.risk_level 获取, LOW/MEDIUM/HIGH badge

## 7. SSE 事件协议

### 7.1 现有事件

```
connected       — 连接成功
step_progress   — 步骤状态变更（RUNNING/COMPLETED）
task_complete   — 任务完成
task_failed     — 任务失败
```

### 7.2 新增事件

```
code_token      — 流式代码 token（从 Redis pub/sub 转发）
                  payload: { type: "code_token", task_id: N, data: "token_text" }
```

前端 `useTaskStream` hook 需要处理新事件类型，分发到对应组件。
