## 1. 产品概述

### 1.1 愿景

让完全不懂代码的产品经理/运营，通过自然语言描述需求，由 AI 驱动 **7 阶段全流程开发流水线** — 从需求澄清、风险识别、测试先行、代码生成、自动化测试、制品管理到 K8s 部署运维 — 在工程规范约束下生成生产级代码，稳定发布到生产环境。

### 1.2 定位

Forge 是一个 **AI 驱动的全流程开发平台** — 不仅仅是 AI 代码生成工具，而是覆盖从需求到部署运维全生命周期的 7 阶段流水线，为 AI Agent 提供规范约束、风险前置、测试先行、机械化验证和持续运维的完整工程化环境。

> **Harness Engineering**（OpenAI 提出）：设计约束环境、反馈循环和基础设施，使 AI 编码代理在规模化场景下可靠运行。Forge 在此基础上进一步实现了 7 阶段端到端覆盖。

本平台是面向生产环境的企业级系统，不是 MVP 原型。分阶段交付的目的是控制范围和节奏，而非降低质量标准。

### 1.3 目标用户

| 角色 | 特征 | 使用方式 |
|------|------|---------|
| 产品经理 / 运营 | 懂业务逻辑但不写代码 | 自然语言描述需求，审批发布结果 |
| 技术管理者 | 负责质量和安全兜底 | 审批高风险变更，配置规范和风险规则 |
| 平台管理员 | 负责平台运维 | 管理平台配置、外部平台接入、紧急停止 |

### 1.4 核心能力

| 能力 | 说明 |
|------|------|
| 从零孵化 | "我要一个优惠券系统" → AI 基于脚手架生成完整新项目 |
| 迭代增强 | "加一个通知渠道" → AI 理解现有架构，在正确位置插入代码 |
| 风险前置 | 需求阶段主动识别技术风险和模糊点，澄清后才进入实施 |
| 测试先行 | 先生成测试用例再写代码，原生框架运行单元/API 测试，Python 统一 E2E |
| 实时预览 | 代码生成过程实时展示文件树变化、Docker 预览环境、Lint 结果 |
| 制品管理 | Docker 镜像构建、OSS 推送、版本语义化管理 |
| K8s 部署 | 自动生成 K8s 资源清单、自动化部署、运维 Agent 持续巡检 |
| 代码浏览 | 关联 GitHub 后在 Web 界面浏览仓库代码（文件树 + 语法高亮），切换分支，查看 AI 分支和 PR |
| 项目记忆 | AI 自动扫描项目生成结构化画像（API/数据模型/架构/业务规则），跨项目关联，增量更新 |
| 云端预览 | PR 级别自动创建临时预览环境，AI 检测项目类型并生成 Dockerfile/K8s 资源，空闲自动回收 |
| 信任分级 | 低风险全自动（生成 → Review → 合并 → 部署），高风险人工审批兜底 |
| AI 方案推荐 | 当存在多种实现方案时，AI 给出结构化推荐（方案对比 + 利弊分析 + 风险评级 + 基于项目上下文的推荐理由），用户点选后 AI 按选定方案执行 |
| 多平台制品 | 支持 Web（Docker→K8s）、移动（Flutter/RN→App Store）、桌面（Tauri/Electron→GitHub Releases）、函数库（npm/PyPI/Go module）的构建和分发 |
| 智能项目接入 | 导入仓库后自动检测项目类型（Web/Mobile/Desktop/API/Library），推荐技术栈和分支策略，引导式配置 |
| 多入口 | Web 工作台 + 钉钉/飞书 IM + CLI，统一后端引擎 |
| 多平台适配 | 支持接入多种代码托管、容器编排、CI/CD 平台，首期对接阿里云生态 + GitHub |

### 1.5 核心理念

**规范即灵魂。** AI 只是执行者，沉淀多年的工程规范才是保证代码质量的根本。小白不需要懂代码，但 AI 必须严格遵守编码规范、安全规范和架构规范。

**测试先行。** 先写测试再写代码。单元测试和 API 测试使用目标项目原生框架（Go→go test, Java→JUnit, JS→Jest），E2E/集成测试统一使用 Python（pytest + Playwright），保证测试与业务代码深度绑定。

**风险前置。** 需求阶段 AI 主动提问澄清模糊点、识别技术风险、检测技术栈约束，在编码之前消除不确定性。

**Harness Engineering 三大支柱**：

| 支柱 | 说明 | Forge 对应 |
|------|------|-----------|
| **Context Engineering** | AI Agent 只能看到仓库内的内容 — 文档、规范、代码必须版本化存在仓库中 | 规范中心 + 项目画像 + Prompt 模板 |
| **Architectural Constraints** | 机械化强制执行架构约束，不依赖文档自觉遵守 | Linter 引擎 + 结构测试 + AI Review + 质量门禁 |
| **Entropy Management** | AI 忠实复制模式（包括坏模式），需要持续扫描和修复代码熵增 | 熵管理后台 + 自动修复 + 质量趋势 |

**反馈速度决定 Agent 自治能力**：

| 反馈层级 | 速度 | 机制 |
|---------|------|------|
| 工具调用后验证 | 毫秒级 | PostToolUse 钩子 |
| 提交前检查 | 秒级 | Pre-commit linter + 结构测试 |
| CI 流水线 | 分钟级 | 构建、测试、安全扫描 |
| 运行时反馈 | 小时级 | 可观测性数据回传 |
| 人工审查 | 天级 | 仅处理新架构决策和高风险变更 |

### 1.6 产品结构

```
Forge Platform — 7 阶段 AI 全流程开发平台
├── Phase 1: 需求澄清与风险识别 ─ 主动提问 + 风险检测 + 技术栈约束
├── Phase 2: 需求规划与任务拆分 ─ DAG 任务分解 + 需求追溯 + 优先级估算
├── Phase 3: 测试用例编写 ─────── 测试先行 + 原生框架 + Python E2E
├── Phase 4: 代码编写与实时预览 ─ 约束生成 + 实时文件树 + Docker + Lint
├── Phase 5: 自动化测试 ────────── 严格顺序执行 + 阻断门禁
├── Phase 6: 制品管理 ─────────── Docker 构建 + OSS 推送 + 版本管理
├── Phase 7: K8s 部署与运维 ───── 资源生成 + 自动部署 + 运维 Agent
│
├── 支撑系统
│   ├── AI 引擎 ————— 中央大脑：多模型路由 + LangGraph 编排
│   ├── 项目画像引擎 —— 全量项目记忆：API/数据模型/架构/业务规则结构化存储 + 向量化索引
│   ├── 鉴权中心 ———— 统一身份认证 + 权限管理
│   ├── 规范中心 ———— 编码规范 + Linter 规则 + 脚手架模板 + Prompt 模板
│   ├── 约束引擎 ———— Linter 管理 + 结构测试 + Agent 可消费错误信息
│   ├── Web 工作台 ——— 用户主界面：需求输入 + 任务看板 + 代码浏览 + 审批 + 预览
│   ├── 代码浏览器 ——— 仓库文件树 + 语法高亮 + 分支切换 + PR 查看
│   ├── 云端预览 ———— PR 级临时环境自动创建/回收 + AI 生成 Dockerfile/K8s
│   ├── IM 机器人 ——— 钉钉/飞书入口，对话式需求提交
│   ├── 实时网关 ———— AI 流式输出 + 进度推送（SSE）
│   ├── 外部平台适配器 — 代码托管 / 容器编排 / CI/CD 的多平台抽象层
│   └── 产品加速库 ——— AI 孵化产品的可选组件库（登录/支付/订单等）
│
├── 基础设施层
│   ├── K8s 集群管理 ─── 单集群多 namespace 隔离
│   ├── 容器化任务执行 ── K8s Job per task（测试/构建/部署）
│   ├── Git 托管混合模式 ─ GitHub App + OAuth 双轨
│   └── 制品存储 ──────── ACR（镜像）+ OSS（产物）
```

### 1.7 一句话端到端流程

```
需求澄清 → 风险识别 → 任务拆分(DAG) → 测试用例先行 → 约束代码生成 → 自动化测试(阻断门禁) → 制品构建(Docker+OSS) → K8s 部署 → 运维 Agent 巡检
```

## 2. 功能需求

### 2.1 AI 引擎

AI 引擎是平台的中央大脑，驱动 7 阶段全流程。

#### 2.1.1 需求澄清与风险识别（Phase 1）

| 功能 | 说明 |
|------|------|
| 自然语言理解 | 接收用户的自然语言需求描述，理解意图 |
| **主动提问澄清** | 需求模糊时 AI 主动提问，生成结构化澄清问题列表，而非被动等待 |
| **技术风险识别** | AI 在需求阶段即识别潜在技术风险（性能瓶颈、安全隐患、兼容性问题） |
| **技术栈约束检测** | 自动检测目标项目技术栈，确保生成方案与现有架构兼容 |
| 需求确认 | 向用户展示 AI 理解的需求 + 风险清单，用户确认后才继续 |

#### 2.1.2 需求规划与任务拆分（Phase 2）

| 功能 | 说明 |
|------|------|
| **DAG 任务分解** | 将需求拆解为有向无环图（DAG）结构的技术任务，标识依赖关系 |
| **需求追溯** | 每个任务可追溯到原始需求条目，保证完整覆盖 |
| **优先级估算** | AI 基于依赖关系和复杂度自动估算任务优先级和执行顺序 |
| 任务清单生成 | 输出结构化任务清单（改哪些模块、新增哪些接口、数据库变更） |

#### 2.1.3 代码生成（Phase 4）

| 功能 | 说明 |
|------|------|
| **约束代码生成** | 在规范约束下生成代码，实时校验 Lint 规则和架构约束 |
| 上下文感知 | 基于项目已有代码、规范、Schema 生成代码，确保与现有架构一致 |
| **实时文件树** | 生成过程中实时展示文件树变化，用户可看到新增/修改的文件 |
| **Docker 预览** | 生成完成后自动构建 Docker 预览环境，支持实时预览效果 |
| 多文件原子提交 | 一次任务生成的多个文件作为一个原子提交 |
| 独立分支 | AI 在独立分支（ai/feature-xxx）上工作，不直接操作主分支 |

#### 2.1.4 AI 代码审查

| 功能 | 说明 |
|------|------|
| 编码规范检查 | 对照规范中心规则，检查代码是否合规 |
| 安全扫描 | OWASP 规则 + SQL 注入 + XSS 检查 |
| 逻辑一致性 | 检查生成代码是否自洽、接口是否匹配 |
| 评分报告 | 输出评分（0-100）+ 问题列表 + 修复建议 |
| 自动修复 | 有问题时 AI 自动修复，最多 3 轮，超限升级人工 |

#### 2.1.5 多模型支持

| 功能 | 说明 |
|------|------|
| 多模型路由 | 按任务复杂度选择合适模型（高复杂度用强模型，简单任务用轻量模型） |
| 故障降级 | 主模型不可用时自动切换到备选模型（Claude → GPT → 通义） |
| 模型可插拔 | 新增 AI 模型不需要修改核心逻辑 |

#### 2.1.6 风险评估

| 功能 | 说明 |
|------|------|
| 两阶段评估 | 方案规划后初评 + AI 审查后终评，取较高值 |
| 风险分级 | 低风险自动流转，高风险进入人工审批 |
| 评估维度 | 需求类型、影响范围、文件数、Review 评分、是否涉及安全/支付模块 |

#### 2.1.7 并发冲突处理

| 功能 | 说明 |
|------|------|
| 独立分支隔离 | 每个任务独立分支，不直接操作 develop |
| 冲突检测 | 提交前检查目标分支状态，冲突时 AI 尝试自动解决（最多 1 次） |
| 人工兜底 | 自动解决失败时通知人工，**绝不强制覆盖人工代码** |

#### 2.1.8 DB 迁移安全

| 功能 | 说明 |
|------|------|
| 可回滚迁移 | 必须同时生成 UP 和 DOWN 迁移脚本 |
| 临时环境验证 | 先在临时环境执行验证可逆性 |
| 破坏性操作拦截 | DROP TABLE/COLUMN 等操作强制人工审批 |

### 2.2 测试体系

**测试先行**是 Forge 的核心理念之一：先生成测试用例，再生成业务代码。

#### 2.2.1 测试用例编写（Phase 3 — 先于代码）

| 功能 | 说明 |
|------|------|
| **测试先行** | 在代码生成之前，AI 先生成测试用例，定义预期行为 |
| **原生框架单元测试** | 单元测试使用目标项目的原生测试框架（Go→go test, Java→JUnit 5, JS→Jest） |
| **原生框架 API 测试** | API/接口测试同样使用原生框架，与项目技术栈深度绑定 |
| **Python E2E 测试** | E2E/集成测试统一使用 Python（pytest + Playwright），跨技术栈通用 |
| 测试用例覆盖 | 正常路径 + 异常路径 + 边界条件，AI 自动生成全面覆盖 |

#### 2.2.2 自动化测试执行（Phase 5）

| 测试层 | 内容 | 工具 | 执行方式 |
|--------|------|------|---------|
| 单元测试 | AI 生成的业务代码单测 | 原生框架（go test / JUnit 5 / Jest） | **严格顺序执行，失败阻断** |
| API/接口测试 | 根据 API 定义生成测试用例 | 原生框架 | **严格顺序执行，失败阻断** |
| 集成测试 | 跨服务的业务流程测试 | Python（pytest） | 可并行执行 |
| E2E 测试 | 端到端用户流程测试 | Python（pytest + Playwright） | 可并行执行 |
| 回归测试 | 运行项目已有的全部测试用例 | 原生框架 | 合并前执行 |

**阻断门禁规则**：任一测试层失败 → AI 分析失败原因 → 自动修复（最多 3 轮）→ 超限升级人工。上一层未通过不得进入下一层。

#### 2.2.3 质量门禁

| 检查项 | 要求 |
|--------|------|
| 编译通过 | 必须 |
| 单测覆盖率 | ≥ 配置阈值（默认 60%，可按项目调） |
| AI Review 通过 | 必须 |
| 安全扫描无高危 | 必须 |
| 镜像漏洞扫描 | 无 Critical 级别 |
| API 兼容性 | 不允许破坏性变更（接口删除/改签名） |

任一不通过 → 阻断部署 + AI 自动修复。

### 2.4 代码托管与 Git 操作

#### 2.4.1 Forge 托管仓库（新项目）

| 功能 | 说明 |
|------|------|
| **GitHub App 托管** | Forge 注册为 GitHub App，安装到 `forge-managed` Organization，自动创建和管理仓库 |
| **自动创建仓库** | 用户创建项目时 Forge 自动创建 `forge-managed/{tenant-slug}-{project-slug}` 仓库 |
| **零门槛接入** | 用户无需自己的 GitHub 账号，通过 Forge Web 界面即可管理代码 |
| **Installation Token** | 使用 GitHub App Installation Token 操作仓库，自动刷新，不依赖个人账号 |

#### 2.4.2 用户仓库关联（已有项目）

| 功能 | 说明 |
|------|------|
| **个人 OAuth 授权** | 用户通过 OAuth 授权 Forge 访问个人仓库，Token 存储在 `user_identities` 表 |
| **企业 GitHub App 安装** | 企业管理员将 Forge GitHub App 安装到企业 Org，支持精确控制仓库权限 |
| **双模式自动选择** | 操作仓库时自动选择最优 Token：优先 Installation Token，降级到 OAuth Token |
| **团队稳定性** | Installation Token 不依赖个人账号，团队成员离职不影响平台操作 |

#### 2.4.3 自动推送与 Token 管理

| 功能 | 说明 |
|------|------|
| **自动推送** | AI 生成代码后自动推送到用户账号/企业 Org 的 GitHub 仓库 |
| **GitHub App JWT** | Forge 启动时用 App 私钥签名生成 JWT（10min 有效），用于获取 Installation Token |
| **Token 自动刷新** | Installation Token（1h 有效）缓存在 Redis，过期前 5 分钟自动刷新 |
| **多 Installation 隔离** | 每个 Organization 安装独立 Token，互不影响 |

### 2.5 DevOps 自动化

#### 2.5.1 任务容器（K8s Job）

| 功能 | 说明 |
|------|------|
| **K8s Job 隔离** | 每个任务步骤（测试/构建/部署）在独立 K8s Job 容器中执行，互不干扰 |
| **资源限制** | 单任务容器限制 CPU 2 核 / 内存 4Gi，防止资源滥用 |
| **超时控制** | 任务容器 30 分钟超时（activeDeadlineSeconds），防止僵死任务 |
| **自动清理** | 任务完成后 5 分钟自动清理容器（ttlSecondsAfterFinished） |
| **forge-jobs namespace** | 所有任务容器统一部署在 `forge-jobs` namespace，与平台服务隔离 |
| **租户标签** | 每个 Job 打上 `forge.io/tenant` 和 `forge.io/task` 标签，便于追踪和审计 |

#### 2.5.2 K8s 部署（Phase 7）

| 功能 | 说明 |
|------|------|
| **K8s 资源生成** | AI 自动生成 Deployment / Service / Ingress / ConfigMap 等 K8s 资源清单 |

| **自动扩缩容** | 基于负载指标自动 HPA 扩缩容 |
| **回滚能力** | 一键回滚到上一版本，自动回滚触发规则 |

#### 2.5.3 灰度部署

| 功能 | 说明 |
|------|------|

#### 2.5.4 环境管理

### 2.10 规范中心

#### 2.10.1 定位

规范中心是 AI 代码生成质量的根本保障，不只是静态文件，而是可管理、可演进、可度量的规范服务。

#### 2.10.2 内容结构

| 内容 | 说明 |
|------|------|
| 编码规范 | 按语言/框架组织（Go、Java、TypeScript、SQL、Redis、API 设计、安全编码、命名规范、Git 工作流） |
| 脚手架模板 | Go 微服务骨架、Java Spring Boot 骨架、Next.js 前端骨架、Python FastAPI 骨架（按目标项目技术栈） |
| AI Review 规则 | 编码规约、安全（OWASP Top 10）、性能、数据库规范、API 兼容性、团队自定义 |
| Prompt 模板 | 核心模板（详见 2.10.4） |
| 规范测试套件 | bad-code-samples + good-code-samples + eval 测试 |

#### 2.10.3 编码规范基线

以已有工程实践沉淀为基线（详见 [references/coding-standards.md](references/coding-standards.md)）。规范按目标项目技术栈组织，核心要点示例：

| 规范维度 | Go 项目 | Java 项目 |
|---------|---------|----------|
| 领域模型命名 | Model / DTO / Response 区分 | DO / DTO / VO / BO 严格区分 |
| 响应封装 | 统一 `Result[T]` 泛型包装 | 统一 `Result<T>` 包装 |
| 错误处理 | 显式 error 返回，自定义 AppError 类型 | 分层异常：BizException / SysException |
| 错误码 | 集中式 ErrorCode 常量，禁止硬编码 | 集中式 ErrorCode 枚举，禁止硬编码 |
| 依赖注入 | 构造器注入（Deps struct） | 构造器注入，禁止字段注入 |

#### 2.10.4 Prompt 模板体系

**Prompt 是平台最核心的资产，必须在开发前完成设计、评审和 eval 测试。**

每个 Prompt 调用由四层叠加：

| 层级 | 内容 | 变化频率 |
|------|------|---------|
| System Prompt（固定层） | AI 角色定义、行为约束、输出格式 | 极少变化 |
| Standards Injection（项目层） | 编码规范、技术栈约束、命名规则 | 按项目不同 |
| Context（任务层） | 相关代码、Schema、API 契约、变更历史 | 每次任务 |
| User Input（用户层） | 需求描述、澄清回复、修改要求 | 每次交互 |

**核心 Prompt 模板**：

| 模板 | 用途 | 输入 | 输出 |
|------|------|------|------|
| requirement-analysis | 需求澄清 + 风险识别 | 用户需求 + 项目上下文 | 澄清问题 + 风险清单 + 任务 DAG |
| test-generation | 测试先行生成 | 任务清单 + API 定义 + 规范 | 测试用例代码（原生框架 + Python E2E） |
| code-generation | 约束代码生成 | 任务清单 + 测试用例 + 相关代码 + 规范 | 可提交的代码文件 |
| code-review | 代码审查 | 生成代码 + 规范 + Review 规则 | 评分 + 问题 + 修复建议 |
| fix-generation | 修复生成 | Review/测试反馈 + 原始代码 + 规范 | 修复后的代码 |
| doc-generation | 文档生成 | 代码 + API 定义 | API 文档 + 变更日志 |

**质量保障**：每个 Prompt 配套 eval 测试，修改后与上一版本 A/B 对比确保不退化。

#### 2.10.5 规范继承机制

```
公司级规范（默认）→ 团队级覆盖（可选）→ 项目级覆盖（可选）→ 最终生效规范
```

#### 2.10.6 规范效果度量

持续追踪：AI 生成代码合规率、AI Review 准确率、自动合并代码线上故障率、Prompt 生成质量评分。形成反馈闭环。

### 2.17 版本管理与多需求并行

> **Phase 2 交付**（版本 CRUD + 并发冲突检测 + 顺序合并），**Phase 3 增强**（Feature Flag + 灰度发布联动）

#### 2.17.1 定位

真实项目以版本迭代推进（v1.2 加三个功能），多个需求同时开发。Forge 需要在版本维度组织需求，自动检测并发冲突，保证合并安全。

#### 2.17.2 版本迭代工作流

| 步骤 | 操作 | 说明 |
|------|------|------|
| 创建版本 | PM 输入版本号 + 描述 | 如 "v1.2 — 用户积分相关功能" |
| 添加需求 | 在版本下创建多个任务 | 每个任务独立进入 AI 流水线 |
| 并行开发 | 无冲突的任务并行执行 | 系统自动检测文件级冲突，有冲突则排队 |
| 顺序合并 | 先完成先合并 + 自动 rebase | 合并前 rebase 到最新 main，冲突时 AI 自动解决（1 次） |
| 版本发布 | 所有任务完成 → 打 tag → 部署 | PM 一键操作 |

#### 2.17.3 并发冲突检测

| 检测层 | 时机 | 方法 | 动作 |
|--------|------|------|------|
| 规划期 | PlannerAgent 输出 touched_files 后 | 文件列表交集 | 阻塞后置任务或标记警告 |
| 生成期 | 代码生成完成后 | git merge-tree 模拟 | 暂停 + 通知 |
| 合并期 | PR 准备合并时 | GitHub API 冲突检查 | AI rebase（1 次）→ 失败通知技术管理者 |

#### 2.17.4 版本号规则

- PM 指定 major.minor（如 v1.2），patch 自动递增
- AI 生成的 commit 遵循 Conventional Commits（feat: / fix: / refactor:）
- 版本发布时自动打 git tag

#### 2.17.5 数据结构

```

  id, tenant_id, project_id, version, status, description,
  released_at, created_at, updated_at
  （关联任务通过 tasks.version_id 外键查询，不冗余存储）

status 枚举:
  PLANNING → IN_PROGRESS → TESTING → RELEASED
                                   → CANCELLED

Task 扩展字段（并发协调）:
  version_id        — 关联的版本
  conflict_status   — NONE / DETECTED / WAITING / RESOLVED
  blocked_by        — 阻塞当前任务的 task_id 列表
  touched_files     — AI 预计修改的文件列表（PlannerAgent 输出）
```

#### 2.17.6 API 设计

| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 创建版本 | POST | `/api/projects/{id}/versions` | 输入版本号 + 描述 |
| 版本列表 | GET | `/api/projects/{id}/versions` | 含任务进度统计 |
| 版本详情 | GET | `/api/projects/{id}/versions/{vid}` | 含所有任务状态 |
| 更新版本 | PUT | `/api/projects/{id}/versions/{vid}` | 修改描述/状态 |
| 发布版本 | POST | `/api/projects/{id}/versions/{vid}/release` | 打 tag + 触发部署 |

### 2.18 AI Harness Engineering 基座

> **Phase 2 交付**（上下文缓存 + Agent Loop + 上下文工具），参考 [learn-claude-code](https://github.com/shareAI-lab/learn-claude-code) Harness Engineering 范式

#### 2.18.1 定位

AI 代码生成的质量取决于上下文工程。当前架构将所有上下文（规范、画像、规则）一次性塞入 system prompt，导致信噪比低、token 浪费、无法精准获取信息。

Harness Engineering 范式：**让 Agent 按需获取上下文，而非预加载所有内容。**

#### 2.18.2 上下文工具

AI Agent 在生成代码时，可通过工具主动查询项目信息：

| 工具 | 用途 | 典型场景 |
|------|------|---------|
| query_api_catalog | 查询现有 API | 生成新 API 前，查看命名和参数风格 |
| query_db_schema | 查询表结构 | 生成数据模型时，了解已有字段和索引 |
| query_business_rules | 查询业务规则 | 避免生成违反业务约束的代码 |
| query_module_graph | 查询模块依赖 | 确定 import 路径和模块边界 |
| read_project_file | 读取源码文件 | 参考现有代码风格，保持一致性 |

#### 2.18.3 Agent Loop

每个 AI Agent 从单轮调用改为多轮工具循环（最多 5 轮），支持在推理过程中主动获取所需信息。编码规范始终在 system prompt 中（核心约束不延迟加载），画像等大体量数据移到工具按需查询。

#### 2.18.4 上下文缓存

同一工作流内的多个 Agent（分析、规划、测试、生成、Review）共享一份缓存的项目上下文，避免重复获取。

### 2.19 AI 方案推荐系统

> **Phase 2 交付**（方案推荐卡片 + 上下文感知推荐），详见 [SP-2-ai-recommendation.md](plans/SP-2-ai-recommendation.md)

#### 2.19.1 定位

当 AI 在需求分析或任务规划中识别出多种可行方案时，不应沉默选择或模糊处理，而应结构化展示所有方案的利弊，基于项目上下文给出推荐，让用户做出知情决策。

#### 2.19.2 推荐触发场景

| 场景 | 示例 | 推荐内容 |
|------|------|---------|
| 架构选择 | "加积分功能" — 在现有服务扩展 vs 新建独立服务 | 方案对比 + 基于项目规模的推荐 |
| 技术栈选择 | "做一个前端项目" — Next.js vs Nuxt vs SvelteKit | 框架对比 + 基于团队经验的推荐 |
| 任务拆分策略 | 3 个子任务 — 串行稳妥 vs 并行高效 | 风险对比 + 基于依赖关系的推荐 |
| 部署策略 | "部署到生产" — 直接部署 vs 灰度 vs 蓝绿 | 策略对比 + 基于风险等级的推荐 |

#### 2.19.3 推荐卡片结构

每个推荐包含 2-3 个方案，每个方案有：
- 方案标题 + 简要描述
- 优势列表（绿色）
- 劣势列表（红色）
- 风险等级（LOW / MEDIUM / HIGH）
- 是否为 AI 推荐（带推荐理由）
- 推荐的上下文依据（如 "项目当前有 10 个 API，UserService 有 3 个方法"）

用户点选方案后，AI 按选定方案继续执行。

#### 2.19.4 上下文感知推荐

AI 推荐不是通用建议，而是基于当前项目的具体情况：
- 项目画像（API 数量、模块复杂度、技术栈）
- 编码规范（团队已有的架构约束）
- 历史任务（之前类似需求是怎么做的）

### 2.20 多平台项目支持

> **Phase 2 交付**（项目类型检测 + 构建模板 + 分支策略适配），详见 [SP-1-project-onboarding.md](plans/SP-1-project-onboarding.md) 和 [SP-3-multi-platform-artifacts.md](plans/SP-3-multi-platform-artifacts.md)

#### 2.20.1 定位

不同项目类型（Web/移动/桌面/函数库）有不同的构建产物、部署目标和发布节奏。Forge 应在项目创建时自动识别类型，并相应配置构建流水线和分支策略。

#### 2.20.2 支持的项目类型

| 类型 | 子类型 | 构建产物 | 部署目标 | 分支策略 |
|------|--------|---------|---------|---------|
| Web 应用 | Next.js / Nuxt / SPA | Docker 镜像 | K8s / Serverless / CDN | 主干开发（持续部署） |
| 移动应用 | Flutter / React Native | AAB + IPA | App Store + Play Store | 发布列车（Release Train） |
| 桌面应用 | Tauri / Electron | .exe / .dmg / .AppImage | GitHub Releases | GitHub Flow + Release 分支 |
| 后端 API | Go / Java / Python / Node | Docker 镜像 + 二进制 | K8s / 云函数 | 主干开发 |
| 函数库 | npm / PyPI / Go module | Package 包 | Package Registry | GitHub Flow + 自动发布 |
| 全栈单仓 | Monorepo (Nx/Turbo) | 多个产物 | 按子项目配置 | 主干开发 + 影响检测 |

#### 2.20.3 项目类型自动检测

导入仓库后，系统扫描仓库文件树，通过签名文件检测项目类型：
- `next.config.*` → Next.js Web 应用
- `pubspec.yaml` + `flutter` 依赖 → Flutter 移动应用
- `tauri.conf.json` → Tauri 桌面应用
- `go.mod` + `cmd/` 目录 → Go 后端 API
- `turbo.json` 或 `nx.json` → Monorepo

检测结果存入 `projects.tech_stack`（扩展的 JSONB），用户可手动覆盖。

#### 2.20.4 分支策略适配

| 策略 | 适用项目 | 特征 |
|------|---------|------|
| **主干开发 (Trunk-Based)** | Web SaaS、后端 API | 短命功能分支 → main → 自动部署，无 release 分支 |
| **GitHub Flow** | 函数库、简单项目 | 功能分支 → main → 自动发布，语义化版本 |
| **发布列车 (Release Train)** | 移动/桌面应用 | 功能分支 → main → release/{version} → Store 提审，支持热修复分支 |

分支策略是项目级配置，根据检测到的项目类型自动设置，用户可在项目设置中修改。

## 3. 业务流程

### 3.1 核心流程 — 7 阶段 AI 开发流水线

以"给优惠券系统加一个按用户等级发放的功能"为例：

```
① 需求澄清与风险识别（Phase 1）
   用户通过 Web 工作台 / IM 机器人 输入自然语言需求
   → AI 主动提问澄清模糊点（"按等级发放是指不同等级不同面额，还是不同等级不同可领取数量？"）
   → AI 识别技术风险（"涉及优惠券核心发放逻辑，需注意并发扣减安全"）
   → AI 检测技术栈约束（"目标项目为 Go + Gin，数据库 PostgreSQL"）
   → 反馈给用户确认：需求理解 + 风险清单

② 需求规划与任务拆分（Phase 2）
   AI 加载项目上下文（规范 + 代码 + 架构），生成 DAG 任务图
   → 标识任务依赖关系和执行顺序
   → 每个任务追溯到原始需求条目
   → 初步风险评估

③ 测试用例编写（Phase 3 — 先于代码）
   AI 先生成测试用例，定义预期行为
   → 单元测试：项目原生框架（Go→go test，Java→JUnit 5，JS→Jest）
   → API 测试：项目原生框架（Go→httptest，Java→MockMvc）
   → E2E 测试：Python pytest + Playwright
   → 用户确认测试覆盖范围

④ 代码编写与实时预览（Phase 4）
   基于测试用例和规范约束生成实现代码
   → 实时展示文件树变化
   → 实时 Lint 检查
   → Docker 预览环境自动构建
   → 原子提交到独立分支
   → AI Review：编码规范 + 安全扫描 + 逻辑一致性
   → 有问题自动修复（最多 3 轮）
   → 终评风险等级

⑤ 自动化测试（Phase 5）
   严格顺序执行，上一层未通过不进入下一层：
   单元测试(go test/JUnit) → API 测试 → 集成测试(pytest) → E2E 测试(pytest+Playwright) → 回归测试
   → 失败时 AI 分析日志并修复（最多 3 轮）
   → 3 轮未通过升级人工

⑥ 制品管理（Phase 6）
   Docker 镜像构建（多阶段优化）
   → 镜像漏洞扫描
   → 语义化版本号生成
   → 推送到 OSS / 容器镜像仓库
   → 记录制品元数据

⑦ K8s 部署与运维（Phase 7）
   AI 生成 K8s 资源清单（Deployment/Service/Ingress/ConfigMap）
   → 低风险 → 自动部署到 dev/staging
   → 高风险 → 创建 MR → 人工审批后部署
   → 生产 → 必须人工审批 + 灰度发布
   → 运维 Agent 持续巡检：健康检查、资源利用率、异常检测
   → 异常时自动告警或回滚

   全程实时推送进度，用户可在任意环节介入、暂停、取消
```

### 3.2 任务生命周期

```
SUBMITTED → ANALYZING → PLANNING → TEST_WRITING → GENERATING → REVIEWING → TESTING → BUILDING → DEPLOYING → DONE
                                                      │                       │          │
                                                 HUMAN_REVIEW             ROLLBACK   BUILD_FAILED
                                                      │
                                                APPROVED / REJECTED
```

- 任何环节失败可从失败点重试（最多 3 轮，超限升级人工）
- HUMAN_REVIEW 是风险分级关键节点 — 低风险跳过，高风险卡住等人审批
- TEST_WRITING 阶段在 GENERATING 之前，体现测试先行

### 3.3 IM 交互流程

```
用户 @forge + 需求描述
  → 机器人回复结构化理解 + 风险提示，请用户确认
  → 确认后创建任务，实时推送 7 阶段进度卡片
  → 完成后推送结果卡片（评分 + 风险等级 + 制品版本 + 操作按钮）
```

## 4. 业务规则

### 4.1 风险分级规则

**低风险判定**（全部满足才算低风险）：
- 纯配置/文档/测试变更，或新增独立接口
- 影响文件数 ≤ 5
- AI Review 评分 ≥ 90
- 不涉及安全/支付/核心业务逻辑

**高风险**（任一命中）：
- 修改核心业务逻辑 / 数据库 Schema 变更 / 涉及权限/支付/安全
- 影响文件数 > 10
- AI Review 评分 < 90 或有告警

### 4.2 三级紧急停止开关

| 级别 | 名称 | 影响范围 | 触发方式 | 恢复方式 |
|------|------|---------|---------|---------|
| L1 | 暂停提交 | 阻断新任务，已执行任务继续 | Web / IM / API | 原路解除 |
| L2 | 冻结引擎 | 任务暂停，阻断代码提交和部署 | Web / API | 手动恢复 + 逐任务确认 |
| L3 | 全面停机 | 切断所有入口 + 停止所有工作负载 | 仅 API | 双人确认后逐步恢复 |

**自动触发规则**：
- 连续 3 次部署失败且错误率 > 5% → 自动触发 L1
- 1 小时内回滚次数 > 3 → 自动触发 L2

### 4.3 失败处理规则

**原则：禁止静默失败。**

| 失败场景 | 最大重试 | 处理 | 用户通知 |
|---------|---------|------|---------|
| AI 代码生成失败 | 3 轮 | AI 自动修复，超限放弃 | "AI 无法完成，已转交开发团队" + 部分代码 + 失败原因 |
| 测试执行失败 | 3 轮 | AI 分析失败原因并修复代码/测试 | 失败测试用例 + 原因分析 |
| 制品构建失败 | 3 轮 | AI 分析 Dockerfile/构建日志并修复 | 错误日志摘要 |
| 部署失败 | 1 次回滚 | 自动回滚 | "部署失败，已自动回滚" + 日志 |
| 外部平台 API 失败 | 3 次 | 退避重试 | 超限后通知任务暂停 |

每次失败通知必须包含：失败环节、原因描述、已完成的工作、下一步建议。

### 4.4 成本控制规则

| 层级 | 机制 | 说明 |
|------|------|------|
| 任务级 | 预估展示 | 提交前展示预估消耗和费用，用户确认后执行 |
| 任务级 | 硬上限 | 单次任务 Token 上限（默认 1M），超限中止 |
| 租户级 | 月度预算 | 80% 软告警，100% 硬限制 |
| 平台级 | 并发控制 | 全局最大并发 AI 任务数，队列排队 |

### 4.5 多租户隔离规则

- 所有数据按 tenant_id 隔离
- 鉴权配置支持全局默认 + 租户级覆盖
- K8s namespace 隔离: 每个租户独立 `tenant-{id}-{env}` namespace，ResourceQuota 限制资源上限
- 任务容器与平台服务 namespace 隔离，NetworkPolicy 禁止任务容器访问 `forge-system`
- 后续如有大租户需要独立部署，再做更强隔离（独立集群）

### 4.6 并发冲突规则

- AI 永远不会强制覆盖人工代码
- 冲突时优先保护人工变更
- 冲突自动解决失败 → 标记 CONFLICT → 通知人工

## 5. 非功能需求

### 5.1 可用性

| 指标 | 要求 |
|------|------|

| 指标 | 要求 |

| 要求 | 说明 |
|------|------|
| 传输加密 | 全链路 HTTPS / WSS |
| 凭证管理 | 所有密钥和凭证加密存储，定期轮转 |
| AI 安全审查 | 每次生成代码必须经过安全扫描 |
| 数据隔离 | 租户间数据完全隔离 |
| 审计日志 | 所有敏感操作可追溯 |
| 容器安全 | Docker 镜像构建后自动漏洞扫描 |

| **任务容器沙箱** | 每个任务在独立 K8s Job 中执行，30 分钟超时，完成后自动清理 |

### 5.4 可观测性

| 要求 | 说明 |
|------|------|
| Forge 平台自身监控 | 全量服务 metrics + 告警（Grafana + Prometheus） |
| 结构化日志 | 集中采集和检索（Loki） |
| 链路追踪 | 跨服务调用全链路可追溯 |
| AI 用量追踪 | 每次模型调用记录 Token、费用、延迟 |
| **运行时反馈闭环** | 可观测性数据（延迟、错误率、资源消耗）回传 AI 引擎，作为迭代上下文 |
| **部署后验证** | 部署后自动检查运行时健康状态，异常时触发告警或自动回滚 |

### 5.5 可扩展性

| 要求 | 说明 |
|------|------|
| 外部平台 | 新增代码托管/容器编排/CI/CD 平台不需改核心逻辑 |
| AI 模型 | 新增模型只需实现统一接口 |

| 测试框架 | 新增目标语言测试框架只需扩展测试生成模板 |

## 7. 风险分析

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| AI 生成代码质量不达标 | 高 | 高 | 测试先行 + 多轮 Review + 修复循环；规范基线 + Prompt eval |
| AI 生成测试用例覆盖不足 | 中 | 高 | 覆盖率门禁 + AI 自动补充边界用例 + 人工审查 |
| AI 模型 API 不稳定/限流 | 中 | 高 | 多模型 fallback；请求排队；错峰调度 |
| 外部平台 API 限流或不可用 | 中 | 高 | 适配器内置缓存 + 限流 + 重试 |
| 小白用户需求描述模糊 | 高 | 中 | AI 主动提问澄清；风险前置识别；需求模板引导 |
| AI 生成不安全代码 | 中 | 高 | 专项安全 Review 规则；安全模块强制人工审批；安全扫描门禁 |
| Token 费用失控 | 中 | 高 | 任务级预估 + 租户月度预算 + 硬上限中止 |
| AI 生成不可逆 DB 迁移 | 中 | 高 | 必须 UP+DOWN 迁移；临时环境验证可逆性；破坏性操作强制人工审批 |
| 生产事故 | 低 | 极高 | 三级紧急停止开关；灰度 + 自动回滚；运维 Agent 巡检 |
| Docker 镜像安全漏洞 | 中 | 高 | 构建后自动漏洞扫描；Critical 级别阻断部署 |
| K8s 资源配置不当 | 中 | 高 | AI 生成资源清单经模板校验 + 资源限制默认值 |
| 上下文窗口不足 | 中 | 中 | 摘要 + 向量检索 + Token 预算裁剪 |
| 适配器抽象泄漏 | 中 | 中 | 最小公共能力集设计；扩展第二套适配器时验证抽象合理性 |
| 多平台凭证管理复杂度 | 低 | 高 | 统一加密存储；凭证过期监控；适配器健康检查 |
| 多租户数据泄露 | 低 | 极高 | 自动注入 tenant_id；统一隔离机制；定期安全审计 |

## 8. 名词说明

| 术语 | 说明 |
|------|------|
| 7 阶段流水线 | Forge 的核心开发流程：需求澄清 → 任务拆分 → 测试先行 → 代码生成 → 自动化测试 → 制品管理 → K8s 部署 |
| Harness Engineering | OpenAI 提出的工程范式 — 设计约束环境、反馈循环和基础设施，使 AI Agent 在规模化场景下可靠运行 |
| Context Engineering | Harness 的子集 — 优化 AI Agent 能看到的所有上下文（代码、文档、规范、可观测性数据） |
| 测试先行 | 先生成测试用例再生成业务代码，测试定义预期行为，代码实现行为 |
| 原生测试框架 | 目标项目自身的测试框架（Go→go test, Java→JUnit 5, JS→Jest），而非外部测试平台 |
| DAG 任务分解 | 将需求拆解为有向无环图结构的任务，明确依赖关系和执行顺序 |
| 风险前置 | 在需求阶段即识别技术风险，而非等到代码生成后才发现问题 |
| 机械化约束 | 通过 linter、结构测试等确定性工具强制执行架构规范，不依赖 AI 自觉遵守 |
| 熵管理 | 定期扫描和修复代码库中的质量偏移（命名漂移、文档过时、死代码等） |
| 制品 | Docker 镜像及其元数据（版本号、构建时间、源代码 commit、测试/安全扫描结果） |
| 运维 Agent | 部署后持续运行的智能巡检代理，监控健康状态、资源利用率、异常检测 |
| 孵化 | 通过 Forge 平台 AI 从零生成一个新的业务系统 |
| 编排服务 | AI 引擎的核心调度组件，驱动任务状态机 |
| 风险等级 | AI 对代码变更的安全评级，决定是否需要人工审批 |
| 产品加速库 | AI 孵化产品可选引用的通用组件库（登录/支付/订单等） |
| 适配器 | 屏蔽外部平台差异的抽象层，上层调用统一接口 |
| 灰度发布 | 新版本按比例/规则逐步放量的发布策略 |
| Prompt 模板 | 预定义的 AI 提示词结构，保证生成代码的质量和一致性 |
| 紧急停止开关 | 三级应急机制，可快速阻断 AI 代码生成和部署 |
| HUMAN_REVIEW | 高风险变更的人工审批环节 |
| SSE | Server-Sent Events — 服务器向客户端推送实时事件的协议 |
| 项目画像 | AI 自动扫描仓库生成的结构化项目知识（API 清单、数据模型、架构图谱、业务规则等） |
| RAG | Retrieval-Augmented Generation — 检索增强生成，通过向量化索引检索相关上下文注入 AI Prompt |
| pgvector | PostgreSQL 向量扩展，支持嵌入向量的存储和相似度检索 |
| 云端预览环境 | PR 级别自动创建的临时运行环境，用户可在浏览器中直接访问 AI 生成的应用 |
| 代码浏览器 | Forge 内嵌的仓库代码查看器，支持文件树导航、语法高亮、分支切换 |
| 跨项目关联记忆 | 多项目间的知识关联，识别共用库、微服务依赖、变更影响范围 |
