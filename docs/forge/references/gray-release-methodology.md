# 灰度发布方法论

> **来源**: kohinur 灰度发布设计理念提炼
> **用途**: Forge 灰度部署基于 APISIX 网关重新实现此方法论

---

## 1. 核心理念

灰度发布的本质是 **在生产环境中安全地验证新版本**，通过精确控制流量的走向，将风险限制在可控范围内。

---

## 2. 三层决策模型

请求到达时，按优先级从高到低依次匹配：

```
请求进入
    ↓
① 环境分区（最高优先级）
   按环境标签隔离流量，不同环境的服务实例完全隔离
    ↓ 未命中
② 规则匹配（中优先级）
   基于请求属性做条件判断，精确控制哪些流量走新版本
    ↓ 未命中
③ 比例分流（最低优先级）
   按百分比随机分配流量到不同版本
    ↓
负载均衡选择具体实例
```

---

## 3. 环境分区

通过环境标签将服务实例划分为多个分区，流量只会路由到匹配分区的实例。

适用场景：
- 开发/测试/预发/生产环境隔离
- 多团队并行开发时的环境隔离
- 特定客户的专属环境

---

## 4. 规则匹配

基于请求属性做布尔表达式匹配，决定流量走向：

**可匹配的维度**：
- HTTP Header（如 user-id、channel、version）
- Cookie 值
- URL 查询参数
- 自定义属性

**支持的运算符**：
- 等于 / 不等于
- 大于 / 小于 / 大于等于 / 小于等于
- 包含 / 前缀匹配 / 后缀匹配
- 正则表达式
- IN（集合成员判断）

**逻辑组合**：
- AND：所有条件必须满足
- OR：任一条件满足即可

适用场景：
- 内测用户优先使用新版本
- 按地域/渠道分流
- 特定用户 ID 精确路由

---

## 5. 比例分流

按权重百分比将流量分配到不同版本组：

- 支持任意比例配置（如新版本 5%，稳定版本 95%）
- 渐进式放量：5% → 20% → 50% → 100%
- 无匹配的灰度实例时自动回退到稳定版本

适用场景：
- 金丝雀发布（小比例验证新版本）
- A/B 测试（按比例分组对比）

---

## 6. 关键设计原则

| 原则 | 说明 |
|------|------|
| 优雅降级 | 灰度逻辑失败时旁路，走默认负载均衡，不影响正常流量 |
| 动态配置 | 灰度规则可实时变更，无需重启服务 |
| 多级绑定 | 支持实例级（精确到 IP）和分组级（粗粒度）两种绑定 |
| 无侵入 | 灰度决策集中在网关层，业务服务无需感知 |

---

## 7. 在 Forge 中的实现

| 方法论层 | APISIX 实现 |
|---------|------------|
| 环境分区 | 路由级别隔离，不同环境完全独立路由 |
| 规则匹配 | traffic-split 插件的 match.vars 条件表达式 |
| 比例分流 | weighted_upstreams 权重配置 |
| 动态配置 | APISIX Admin API 即时变更 |
| 管理界面 | Forge Web 工作台集成灰度管理页面 |

**相比旧方案的改进**：
- 从 SDK 嵌入式（每个服务都要集成 SDK）改为网关集中式（业务服务零侵入）
- 规则变更从轮询 30s 延迟改为 API 调用即时生效
- 从强绑定 Spring Cloud 改为语言无关（任何后端都可灰度）
